# 产品需求文档（PRD）：软件开发需求收集系统

## 1. 项目概述 ##

**软件开发需求收集系统**是一个面向项目启动阶段的交互式前端应用，旨在系统化捕获业务和技术需求。
核心价值：

- 提高需求收集效率，减少遗漏关键问题。
- 引导不同角色（PM、BA、开发负责人、客户）按逻辑流程完成调研。
- 生成可视化报告，辅助项目决策。

系统通过**动态分支问卷引擎**实现灵活问答，支持**必填验证、跳过逻辑、进度追踪**，并在最终生成**摘要报告与可视化分析**。

------

## 2. 目标用户与使用场景 ##

| 用户角色          | 目标                 | 场景                               |
| ----------------- | -------------------- | ---------------------------------- |
| 项目经理 (PM)     | 快速梳理项目范围     | 在项目前期收集需求，保证关键点完整 |
| 业务分析师 (BA)   | 捕获业务逻辑与优先级 | 针对业务部门开展需求访谈           |
| 客户 / 项目相关方 | 轻松提供信息         | 回答问卷无需掌握技术术语           |
| 开发负责人        | 评估技术可行性       | 查看需求概要与技术约束             |
| 系统管理员        | 保证核心问题被回答   | 设置必填项、导出数据和权限管理     |

------

## 3. 用户故事 ##

- **作为用户**，我希望系统以逻辑流程引导我回答问题，避免被技术术语困扰。
- **作为用户**，我希望可以跳过不确定的问题，并由系统标记待定，以便后续澄清。
- **作为项目相关方**，我希望在完成问卷后查看可视化摘要报告，快速理解项目范围和复杂度。
- **作为管理员**，我希望设置强制回答的问题，确保核心项目要素被明确定义。

------

## 4. 功能需求 ##

### 4.1 动态问卷引擎 ###

- **分节管理**：问卷必须至少分为 3 个逻辑模块（项目核心、技术层面、后勤规划）。
- **分支逻辑**：部分问题的回答将决定后续显示内容。例如：
  - Q1: "主要平台是什么？" → Web → Q2；移动端 → Q3。
- **必填验证**：必填问题未回答无法进入下一步。
- **跳过逻辑**：可选问题允许跳过，系统将自动标记为“待定/待定义”。
- **回答记录与版本控制**：每次提交问卷自动保存版本，支持回溯和审查修改历史。

### 4.2 进度追踪 ###

- **模块级进度**：显示当前模块完成度。
- **整体进度条**：显示整个问卷完成百分比，提供视觉反馈。

### 4.3 报告生成与可视化 ###

- **摘要视图**：按模块汇总答案，并标记“待定/需澄清”问题。
- **可视化分析**：
  - 技术复杂度：雷达图
  - 功能分布：饼图
  - 其他可选分析（例如上线风险等级柱状图）
- **导出功能**：
  - PDF / Excel 导出报告
  - 可选择包含/不包含跳过的问题

### 4.4 技术栈 ###

- **前端框架**：SvelteKit
- **语言**：TypeScript
- **样式**：Tailwind CSS
- **测试**：Vitest
- **图表库**：Apache ECharts
- **状态管理**：Svelte Store

## 5. 问卷结构示例 ##

| 模块            | ID   | 问题                         | 类型      | 必填 | 分支逻辑              |
| --------------- | ---- | ---------------------------- | --------- | ---- | --------------------- |
| **1. 项目核心** | Q1   | 主要平台是什么？             | 单选      | 是   | Web → Q2；移动端 → Q3 |
|                 | Q2   | 需要支持哪些浏览器？         | 多选      | 否   | -                     |
|                 | Q3   | iOS、Android还是两者都支持？ | 单选      | 是   | -                     |
| **2. 技术层面** | Q4   | 是否有偏好的技术栈？         | 布尔      | 是   | 是 → Q5；否 → Q6      |
|                 | Q5   | 列出偏好的技术               | 文本      | 是   | -                     |
|                 | Q6   | 是否需要内容管理系统？       | 布尔      | 否   | -                     |
|                 | Q7   | 预计并发用户数？             | 数值范围  | 否   | -                     |
| **3. 后勤规划** | Q8   | 目标上线日期                 | 日期      | 是   | -                     |
|                 | Q9   | 是否已有UI设计稿？           | 单选      | 是   | 是 → Q10；否 → Q11    |
|                 | Q10  | 上传/链接设计文件            | 文本/文件 | 否   | -                     |
|                 | Q11  | 是否需要用户体验研究服务？   | 布尔      | 否   | -                     |

> **优化点**：增加字段类型说明，支持上传文件与范围输入，提高实际操作可行性。

## 6. UI/UX 设计规范 ##

### 6.1 布局 ###

- **侧边栏**：
  - 显示模块列表、进度状态（完成/未完成/待定）
  - 可点击模块跳转
- **内容区**：
  - 卡片式布局显示问题
  - 导航按钮：返回、跳过、下一步
  - 行内校验提示
- **报告仪表板**：
  - 网格布局显示摘要与 ECharts 图表
  - 高亮显示未回答或跳过问题

## 7. 测试与质量保证 ##

### 7.1 单元测试（Vitest） ###

- **核心测试对象**：`QuestionnaireLogic.ts`
- **测试场景**：
  - 分支逻辑跳转正确性
  - 必填校验阻止未回答问题进入下一步
  - 跳过问题状态记录正确
  - ECharts 数据渲染正确

### 7.2 覆盖率目标 ###

- 核心逻辑组件 ≥ 80%
- UI 渲染/交互组件 ≥ 70%

## 8. 实施计划 ##

1. **初始化**：SvelteKit + TypeScript + Tailwind CSS
2. **状态管理**：Svelte Store，实现问卷答案存储与导航状态
3. **问卷引擎**：JSON 驱动分支逻辑，实现动态跳转
4. **UI 实现**：卡片式问卷，侧边栏进度
5. **报告实现**：ECharts 可视化（雷达图、饼图、柱状图）
6. **测试覆盖**：Vitest 单元测试
7. **迭代优化**：添加导出、文件上传、UX 改进

## 9. 数据结构示例 ##

```typescript
interface Question {
  id: string;
  section: string;
  text: string;
  type: 'single' | 'multi' | 'text' | 'boolean' | 'date' | 'file';
  options?: string[];
  isMandatory: boolean;
  nextId?: (answer: any) => string | null; // 动态分支函数
  defaultAnswer?: any;

}
```
